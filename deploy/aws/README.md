# Emails on operator-owned AWS

This Terraform root configuration deploys Emails into an AWS account controlled
by the operator. It contains no maintainer account IDs, hostnames, ARNs, secrets,
cross-account roles, fleet resources, billing integration, tenant control plane,
or hosted SaaS endpoint.

The deployment is dormant by default:

- API and worker desired counts are `0`;
- the API is private unless public HTTPS is explicitly enabled;
- SES receiving is disabled unless explicitly enabled;
- Terraform can create a receipt rule set, but never activates it;
- secret values are never generated by or passed through Terraform;
- an immutable container digest and the expected AWS account ID are required.

`terraform apply` is intentionally outside the repository release process. The
operator owns the AWS credentials, state backend, image registry, approvals,
mail cutover, and ongoing operations.

## Architecture

```text
operator VPN / VPC                         optional Internet
        |                                        |
  Cloud Map private DNS                  Route53 -> HTTPS ALB
        |                                        |
        +--------------- ECS API ---------------+
                              |
                         RDS PostgreSQL

Internet mail -> SES receipt rule (manual activation)
              -> encrypted S3 raw MIME
              -> encrypted SNS -> encrypted SQS -> ECS worker -> PostgreSQL
                                                   |
                                                   +-> DLQ after repeated failure
```

The API, worker, and migration tasks run in private subnets without public IPs.
The API and worker use separate least-privilege task roles. The migration task
uses a separate schema-owner database secret; the long-running tasks receive
only the application-role database secret.

The current container still exposes legacy `mailery` entrypoint names. They are
used only inside ECS task commands for compatibility with the image at this
repository revision; no Mailery-hosted URL or service is configured. Replace
those commands with the final `emails` entrypoints when the renamed image ships.

The separate object bucket is wired as the runtime archive bucket. At this
revision, the inbound worker still keeps raw MIME in the inbound bucket and
stores parsed attachment metadata in PostgreSQL; it does not yet materialize
each parsed attachment into the object bucket. That runtime offload must be
implemented and tested before claiming independent attachment-object durability.

## Cost warning

Dormant does **not** mean free. A default apply creates billable resources even
while ECS desired counts are zero, notably:

- one Multi-AZ RDS PostgreSQL instance;
- NAT gateways only after `enable_nat_gateway = true` (two by default then);
- KMS, S3, SQS/SNS, logs, and monitoring usage;
- an ALB and ACM-related resources when the public endpoint is enabled.

Set `single_nat_gateway = true` only after accepting the loss of AZ-level egress
resilience. Set `db_multi_az = false` only for non-production environments. Get
an AWS Pricing Calculator estimate for the chosen region before applying.

## Prerequisites

- Terraform `>= 1.9, < 2.0`.
- AWS credentials obtained through an operator-controlled SSO session or role.
  Do not put static credentials in Terraform files.
- An operator-owned S3 state bucket with versioning, encryption, public access
  blocked, and restricted access.
- An operator-built Emails image in a registry the ECS execution role can pull.
  `container_image` must include an immutable `@sha256:` digest.
- A region with enough availability zones. If receiving through SES, separately
  verify that SES email receiving is supported in the selected region.

## 1. Configure state and a dormant plan

Copy the examples to private files outside version control:

```bash
cp examples/backend.hcl.example /secure/path/emails-backend.hcl
cp examples/minimal.tfvars.example /secure/path/emails.tfvars
terraform init -backend-config=/secure/path/emails-backend.hcl
terraform plan -var-file=/secure/path/emails.tfvars -out=/secure/path/emails.tfplan
```

Review the saved plan from the correct AWS account. The provider uses
`allowed_account_ids`, and the required `expected_account_id` is checked again
against STS. A wrong account is a hard provider failure.

Apply only through the operator's normal approval process. A first apply creates
infrastructure and empty secret containers, but starts no application tasks.

## 2. Bootstrap PostgreSQL without putting passwords in state

RDS generates and stores the master password in Secrets Manager. Terraform
outputs its secret ARN for bootstrap only. Never inject that master secret into
an ECS task.

Set `database_admin_security_group_ids` to the security group of an
operator-controlled, SSM-managed administration host, then connect from that
host. Remove the temporary grant after bootstrap if it is no longer needed.
From that controlled session with private network access to RDS:

1. Connect using the RDS-managed master secret.
2. Generate independent passwords through an approved secret-generation tool.
3. Create a schema-owner role for migrations and a DML application role.
4. Make the schema owner own the database/schema and grant the app role only
   connect, usage, and table/sequence DML permissions.
5. Store two TLS PostgreSQL URLs in the empty Secrets Manager containers shown
   by `migration_database_url_secret_arn` and `database_url_secret_arn`.

An example privilege shape follows. Replace role names as needed and supply
passwords without copying them into shell history, Terraform, logs, or tickets:

```sql
CREATE ROLE emails_migrator LOGIN;
CREATE ROLE emails_app LOGIN;

ALTER DATABASE emails OWNER TO emails_migrator;
ALTER SCHEMA public OWNER TO emails_migrator;

GRANT CONNECT ON DATABASE emails TO emails_app;
GRANT USAGE ON SCHEMA public TO emails_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO emails_app;
GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA public TO emails_app;
ALTER DEFAULT PRIVILEGES FOR ROLE emails_migrator IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO emails_app;
ALTER DEFAULT PRIVILEGES FOR ROLE emails_migrator IN SCHEMA public
  GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO emails_app;
```

Use `sslmode=require` (or stricter certificate verification supported by the
runtime) in both URLs. Treat secret values as operator data: populate and rotate
them outside Terraform. This repository deliberately contains no
`aws_secretsmanager_secret_version` resources because Terraform would retain
the plaintext in state.

Populate `api_signing_key_secret_arn` with an independently generated,
high-entropy signing key through the same protected workflow.

## 3. Run the one-shot migration

Set `enable_nat_gateway = true` and apply that network-only change while both
service desired counts remain zero. The private migration task needs egress to
pull its image and read Secrets Manager; the application activation gate also
requires this explicit acknowledgement.

Use the output task definition, private subnets, and task security group to run
one Fargate task. The exact AWS CLI invocation depends on the operator profile
and change-control wrapper; its network settings are the Terraform outputs:

- cluster: `ecs_cluster_name`;
- task definition: `migration_task_definition_arn`;
- subnets: `private_subnet_ids`;
- security group: `ecs_task_security_group_id`.

Wait for the task to stop, require exit code `0`, and review the migration log
group `/ecs/<name>/migration`. Then confirm `/ready` can see no pending
migrations before setting `migrations_complete = true`.

## 4. Activate the private service

Set all of the following in the private tfvars file:

```hcl
secrets_ready               = true
migrations_complete         = true
enable_nat_gateway          = true
alarm_notification_topic_arn = "arn:aws:sns:REGION:ACCOUNT:operator-alerts"
api_desired_count           = 2
worker_desired_count        = 1
```

Terraform blocks non-zero services unless all readiness controls are
present. The notification topic must be operator-owned and have a confirmed
subscription.

With the default private endpoint, clients connected through an operator VPN,
Transit Gateway, Direct Connect, or controlled VPC access can resolve:

```text
http://api.emails.internal:8080
```

The task security group accepts that port only from the VPC CIDR. Cloud Map is
not public DNS. For production user access, terminate TLS in an operator-owned
private proxy/load balancer or explicitly enable the HTTPS public endpoint.
ECS Exec is disabled by default and is intended for audited diagnostics, not as
an application access path.

## 5. Optional public HTTPS

Public exposure remains opt-in. Set:

```hcl
enable_public_endpoint = true
service_domain         = "emails.example.com"

# Choose one certificate path:
certificate_arn  = "arn:aws:acm:REGION:ACCOUNT:certificate/ID"
# or create_certificate = true with an operator-owned hosted_zone_id

create_route53_records = true
hosted_zone_id         = "OPERATOR_ZONE_ID"
```

The ALB redirects HTTP to HTTPS, uses a modern TLS policy, drops invalid
headers, and checks `/ready`. API authentication remains required. Review WAF,
rate limits, access logging, and organizational ingress controls before making
the endpoint public.

## 6. SES outbound prerequisites

Set `email_domain` to create an SES identity and DKIM tokens. Route53 records
are created only when both `create_route53_records` and `hosted_zone_id` are
set. Otherwise publish the output tokens through the operator's DNS provider.

Before real sending:

1. Confirm identity and DKIM verification.
2. Publish SPF, DMARC, and a custom MAIL FROM policy appropriate to the domain.
3. Request SES production access and review sending limits, bounce handling,
   complaint handling, suppression policy, and recipient consent.
4. Test with controlled recipients before marking the domain outbound-ready.

The task role can send only through the configured `email_domain` identity. It
does not receive wildcard SES account permissions.

## 7. SES inbound cutover

Set `enable_ses_inbound = true` and provide `inbound_recipients`. Terraform then
creates a new rule set and a store-and-notify rule, but intentionally does not
activate the rule set because activation changes account-global mail routing.

Before activation:

1. Confirm the region supports SES receiving.
2. Confirm SES identity/DNS ownership and that existing MX records are not
   serving another provider.
3. Run the worker and prove a synthetic SES notification travels through SNS,
   SQS, S3, parsing, PostgreSQL, and the user-visible mailbox.
4. Prove retry behavior, DLQ alarming, and a documented replay procedure.
5. Activate the output `ses_receipt_rule_set_name` in a separate reviewed
   operator action.
6. Publish the `ses_inbound_mx_value` only during the approved DNS cutover.

Disabling this Terraform variable does not undo DNS or an already active SES
rule set. Treat rollback as an explicit mail-routing operation.

The receipt rule intentionally does not set SES client-side `kms_key_arn`.
Instead, the bucket enforces transparent SSE-KMS so the JavaScript worker can
read normal MIME bytes through `GetObject`. SES and SNS KMS grants are limited
to this deployment's exact receipt-rule and topic ARNs.

## Operations and recovery

- `/health` proves the process responds; `/ready` is the deployment gate because
  it also checks database connectivity and pending migrations.
- Raw MIME is durable in the versioned inbound bucket. Failed worker messages
  remain on SQS and move to the DLQ after the configured receive count.
- Configure alarm subscriptions before scaling. Included alarms cover RDS CPU
  and storage, SQS age/DLQ, task utilization, and optional ALB target health.
- Test RDS point-in-time restore, S3 object restore, secret rotation, DLQ replay,
  and region/account recovery before calling the deployment production-ready.
- RDS deletion protection defaults on; data buckets and secret containers use
  Terraform `prevent_destroy`. Removing the stack requires an explicit,
  reviewed data-retention procedure.

## Local validation (no AWS mutation)

```bash
terraform init -backend=false
terraform fmt -check -recursive
terraform validate
terraform test
```

The native tests use a mocked AWS provider. They prove the default plan starts
zero ECS tasks, creates no public ALB or SES rule, and rejects API activation
without readiness acknowledgements. They do not replace a reviewed plan in the
operator's real AWS account.
